3. Layer-by-Layer Deep Dive

3.1 Network Layer

The network forms the isolated foundation. The VPC uses a strict 3-tier design ensuring that databases and application servers cannot be accessed directly from the public internet.

  • VPC (10.0.0.0/16): 3-tier subnet design across 3 AZs for high availability.
  • Route 53: Global DNS with latency-based routing and health checks.
  • CloudFront: CDN caching static assets with Origin Shield to protect backend servers.
  • ALB: Application Load Balancer performing Layer 7 routing and SSL termination.
resource "aws_vpc" "main" { cidr_block = "10.0.0.0/16" enable_dns_hostnames = true tags = { Name = "ecommerce-production-vpc" } }

3.2 Compute Layer

The compute layer handles all business logic, rendering, and API execution. It scales dynamically based on traffic.

  • EC2 Auto Scaling: Web tier uses Graviton2 (c6g.xlarge) instances. Scales from 50 to 500 instances.
  • ECS/EKS: Containerized microservices (Cart, Checkout) orchestrated via AWS Fargate.
  • Lambda: Serverless functions with 512MB memory and reserved concurrency for recommendation engines.
resource "aws_autoscaling_group" "web_tier" { desired_capacity = 50 max_size = 500 min_size = 50 target_group_arns = [aws_lb_target_group.web.arn] vpc_zone_identifier = [aws_subnet.app_a.id, aws_subnet.app_b.id] }

3.3 Storage Layer

Storage is completely decoupled from compute to ensure durability and stateless web servers.

  • S3: Millions of product images. Lifecycle rules transition older assets to cheaper tiers.
  • EBS: io2 Block Express volumes with 64k IOPS for EC2-hosted legacy databases.
  • EFS: Shared file system for invoice generation microservices.
  • Glacier: Vault lock policies enforce 7-year WORM compliance on transaction logs.

3.4 Database Layer

State and persistent data are managed by purpose-built databases optimized for specific access patterns.

  • Aurora (PostgreSQL): Core transactional database (Orders). 1 writer + 2 readers, up to 64TB.
  • RDS: Multi-AZ deployment for the product catalog (100M+ SKUs).
  • DynamoDB: On-demand capacity for shopping carts. DAX enabled for microsecond reads.
  • ElastiCache (Redis): Cluster mode with 3 shards for session and catalog caching.

3.5 Analytics & Messaging Layers

  • Kinesis: 100 shards processing real-time clickstream data.
  • Redshift: ra3.xlplus nodes analyzing 5 years of historical sales data.
  • SQS: Decouples checkout from order processing using standard queues (10K msg/sec).
  • EventBridge: Routes inventory update events across disparate microservices.

5. Prime Day Scaling Timeline

The platform is designed to handle extreme volatility during major sales events.

  • T-7 Days: Pre-warming ASG warm pools, Aurora reader promotion, and reviewing AWS limits.
  • T-1 Hour: Scale Lambda reserved concurrency; monitor SQS queue depth alarms.
  • T-0 (Sale Starts): Traffic spikes 10x. ALB routing and CloudFront cache warming absorb the initial hit.
  • T+30 Min (Peak Load): ASG hits 500 EC2 instances. Aurora CPU reaches 80%. SQS buffers thousands of concurrent checkout requests to prevent database locks.
  • T+2 Hours: Traffic normalizes. Gradual scale-down policies trigger; SQS queues drain completely; cost optimization resumes.

7. Connectivity Options

  • Site-to-Site VPN: IPsec tunnels providing ~100ms latency at 1.25 Gbps for failover administrative access.
  • Direct Connect (DX): Dedicated 10 Gbps fiber connection providing 50ms latency for secure, high-throughput corporate data transfers.
  • Transit Gateway: Hub-and-spoke topology connecting Production, Staging, and Shared Services VPCs.
